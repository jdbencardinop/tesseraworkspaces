# Tracked Features

| Slug | Title | State | Compatibility |
|------|-------|-------|---------------|
| `archive-worktree` | Add a tws archive command that removes the worktree from disk but keeps the branch reference in stack.yaml. This saves disk space for large repos where worktrees accumulate. A tws restore or tws new with the same branch name should be able to recreate the worktree from the archived branch. Long-term quality of life improvement. | applied | unknown |
| `auto-detect-repos-workspaces` | We should try and see if we are already in a git repo to define if we should use the default workdir or a custom one for the current repo, and in case we are inside of a worktree we can work backwards and make ts work the same in every of the workspaces folders | applied | unknown |
| `checkout-existing-branch` | Allow tws new to check out an existing branch into a feature worktree instead of only creating new branches. Should work even if the branch is currently checked out in the main repo (git worktree add --detach the main repo first, or use a different approach). This enables migrating existing in-progress work into the tesseraworkspaces model. | applied | unknown |
| `cobra-migration` | Migrate the CLI from manual os.Args parsing to cobra for cleaner flag handling, auto-generated help, and built-in shell completion (bash, zsh, fish, powershell). All 10 commands (add, new, open, sync, stack, delete, list, archive, init) plus global flags (--version, --help) need migration. | applied | unknown |
| `config-edit-command` | Add tws config command to edit configuration from the CLI. Subcommands like tws config set agent_command opencode, tws config set use_tmux true, tws config show. Should support --repo flag to edit per-repo config vs global. Avoids users needing to remember YAML syntax. | requested | unknown |
| `cross-worktree-agent-context` | The CLAUDE.local.md at the feature level is a start for shared context across worktrees, but we need a mechanism for agents to advertise design decisions and breaking changes to sibling branches. Add a changes.log or decisions.yaml at the feature level where agents write design decisions that affect sibling branches. Agents in child branches can read these and decide if they need to adapt. This is the cross-worktree semantic communication layer that no existing stacked-diff tool provides. | requested | unknown |
| `delete-feature` | Add a tws delete (or tws rm) command that removes a feature and all its worktrees. Should remove the worktrees via git worktree remove, delete the feature directory from the workspace, and clean up stack.yaml. Branches are left intact in git for now. | applied | unknown |
| `divergent-stack-sync` | The stack.yaml data model should support DAG dependencies from day one (A->B, A->C divergent stacks), but only implement linear sync initially. Divergent sync (rebasing B and C independently from A) is a future feature. The data model cost of supporting DAGs upfront is minimal, and retrofitting it later into a linear-only schema would be painful. | applied | unknown |
| `fix-auto-select-behavior` | When tws open auto-selects a single feature and single branch, it should still print what was selected so the user knows what happened. Verify it's actually opening the worktree and not the current dir. Add a confirmation or at least a clear message before launching the agent. | requested | unknown |
| `fix-initial-claude-session` | When opening a worktree for the first time via tws open, the tmux command runs "claude -c" which fails because there is no existing session to continue. On first open we should run "claude" (or the configured agent command) without the -c flag. Detect whether a session already exists to decide which variant to use. | applied | unknown |
| `fix-open-cwd-after-exit` | When using tws open without tmux, exiting the agent leaves the user at the repo root instead of the worktree dir. syscall.Exec replaces the process so the parent shell cwd is unchanged. Fix by using exec.Command with Dir set, or consider a shell wrapper approach like worktrunk's shell install. The user should end up in the worktree directory after the agent exits. | requested | unknown |
| `go-module-version-fallback` | Make tws --version report the tagged module version when installed with go install github.com/jdbencardinop/tesseraworkspaces/cmd/tws@latest, while preserving ldflag-injected source builds and adding workflow coverage so dev cannot silently ship. | applied | unknown |
| `interactive-open-picker` | When tws open is called without arguments, show an interactive picker (via fzf or built-in) to select a feature and branch. Could also support tws open <feature> with no branch to pick from branches within that feature. Shell completions from cobra should also suggest valid features and branches. | applied | unknown |
| `keep-track-of-stacked-diffs-and-dependencies` | Our sync method currently just rebases every single branch we have a worktree for, but this might not be what we want, we might just need to advance some of the worktrees or take just changes in one worktree, check if we need one worktree per stack or one worktree per feature, we need to implement a better approach to handling a graph of dependencies in the changes | applied | unknown |
| `lightweight-worktree-handler` | We should have a way to work with worktrees that suits our way of modeling our features, in which a feature folder can have many worktrees, so not all worktrees in the repo are in the same folder, worktrunk might do that out of the box, so we should also document how to do that in worktrunk | applied | unknown |
| `list-features-branches` | Add a tws list (or tws ls) command that shows all features and their branches in the current workspace. Could integrate with fzf for interactive selection or shell completions in a future feature. | applied | unknown |
| `per-repo-config` | Add support for a .tws/config.yaml in the repo root for per-repo configuration. This would override global config for that repo. Settings like agent_command, use_tmux, and workspace path could be set per-repo and committed. Deferred from user-defined-workdir. | applied | unknown |
| `pluggable-agent-command` | ts open currently hardcodes "claude -c" as the agent command launched in the tmux session. We should make this configurable so users can use other agents (opencode, aider, cursor, etc) or no agent at all. Add an agent_command field to the global config (~/.config/ts/config.yaml) and support a per-feature override. Also add claude to the README requirements for now, and document how to change the agent command. | applied | unknown |
| `rebrand-to-tesseraworkspaces` | Rename the project from tesseraspaces/ts to tesseraworkspaces/tws to avoid collision with the TypeScript compiler (ts). Update module path, binary name, all imports, Makefile, README, docs, and CI workflow. | applied | unknown |
| `skill-distribution` | Add a tws init command that installs agent skills teaching coding agents how to use tws. Supports Claude Code (.claude/skills/) and GitHub Copilot (.github/prompts/ and copilot-instructions.md). Skills are embedded in the Go binary via go:embed. Project-local install by default. | applied | unknown |
| `tmux-free-mode` | Make tmux optional for tws open. Add a simpler mode that just cd's into the worktree directory and runs the agent command (claude or configured alternative) directly without tmux wrapping. tmux is nice for session persistence but not always needed. Could be a --no-tmux flag or a config option. | applied | unknown |
| `tmux-session-management` | Add tws close command to kill tmux sessions for a worktree. Detect if a worktree has an associated tmux session (via session naming convention). Prevent conflicts between tmux mode and direct mode when a stale tmux session exists. Show tmux session status in tws list. | requested | unknown |
| `user-defined-workdir` | Currently we use a default working dir for everythin under /Users/jbencardino/ts, but we should have a workspace per repo and let the user configure where to set it up | applied | unknown |
| `versioned-builds` | Add --version and --help flags to the tws CLI. Tag builds with the git tag version via ldflags. CI should only trigger binary builds/releases on tag push; tests, lint, and formatting should run on every push to main and PRs. | applied | unknown |
